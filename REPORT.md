## 2. 개발 방식의 근본적 차이

### module_homepage — "대화형 바이브코딩"

```
[개발자] → Claude 채팅에 요구사항 전달
         ← Claude가 코드 블록으로 응답
[개발자] → 코드를 복사하여 에디터에 붙여넣기
         → 파일 생성/수정을 직접 수행
         → firebase deploy 수동 실행
         → 에러 발생 시 에러 메시지를 다시 채팅에 전달
```

- **장점:** 코드 한 줄 한 줄을 직접 확인하며 학습 가능
- **단점:** 복붙 과정에서 실수 발생, 컨텍스트 소진 시 맥락 재설명 필요, 파일 관리 수동

### agents_homepage — "에이전트 자동화"

```
[개발자] → Claude Code에 요구사항 전달
[Claude Code] → 직접 파일 읽기/쓰기/수정
              → 직접 Git 커밋 수행
              → 에러 발생 시 자동 분석 및 수정 시도
              → git push → Vercel 자동 배포
```

- **장점:** 파일 관리 자동화, Git 이력 체계적, 컨텍스트 효율적 사용, 배포 자동
- **단점:** 코드 변경 과정을 면밀히 추적하려면 별도 확인 필요

### 핵심 차이 요약

| 작업 | module_homepage (수동) | agents_homepage (자동) |
|------|----------------------|----------------------|
| 파일 생성/수정 | 개발자가 직접 복붙 | 에이전트가 직접 수행 |
| Git 버전 관리 | 수동 백업 폴더 (backup_independent) | Git 55개 커밋 자동 관리 |
| 배포 | `firebase deploy` 수동 실행 | `git push` → Vercel 자동 배포 |
| 에러 대응 | 에러 메시지를 채팅에 복붙 | 에이전트가 에러 자동 분석 |
| 컨텍스트 관리 | 대화 초과 시 새 대화 + CLAUDE.md로 복구 | 도구 기반 작업 + 자동 압축 |

## 3. 기술 스택 비교

| 계층 | module_homepage | agents_homepage |
|------|----------------|-----------------|
| **프론트엔드** | 순수 HTML + CSS + JavaScript | Next.js 14 (App Router) + React 18 |
| **스타일링** | CSS 파일 (common.css + 모듈별 CSS) | Tailwind CSS 3.4 + 커스텀 테마 |
| **백엔드/DB** | Firebase (Firestore + Auth + Hosting) | 없음 (localStorage 기반) |
| **상태 관리** | 없음 (각 모듈 독립) | React Context API (4개) |
| **라우팅** | HTML 파일 간 이동 (MPA) | Next.js App Router (SPA) |
| **이미지 최적화** | 없음 (일반 `<img>`) | next/image + onError 폴백 |
| **폰트** | 일반 CSS 로드 | next/font (Google Fonts 최적화) |
| **배포** | Firebase Hosting | Vercel (GitHub 연동 자동 배포) |
| **개발 서버** | Live Server 등 | Next.js dev server (핫 리로드) |

### 기술 스택 선택이 가져온 차이

| 영향 | module_homepage (Firebase) | agents_homepage (Next.js + Vercel) |
|------|--------------------------|-----------------------------------|
| **배포 난이도** | 높음 (인증, 설정, 인덱스 수동 관리) | 낮음 (push만 하면 자동) |
| **백엔드 관련 병목** | 5건 중 3건이 Firebase 관련 | 백엔드 없어서 발생 자체 불가 |
| **SEO** | 별도 처리 필요 | Next.js 메타데이터 내장 |
| **성능 최적화** | 수동 (이미지, 폰트 등) | 프레임워크 내장 (Image, Font, Dynamic Import) |
| **컴포넌트 재사용** | 어려움 (HTML 복붙) | 자연스러움 (React 컴포넌트) |

├── auth/                   ← 각 모듈이 독립된 HTML/CSS/JS
│   ├── login.html
│   ├── auth.css
│   └── auth.js
├── product/
│   ├── product-list.html
│   ├── product.css
│   └── product.js
├── price/
├── review/
├── community/
├── estimate/
├── support/
├── admin/
├── order/
├── wishlist/
├── delivery/
├── promotion/
└── membership/
    총 13개 모듈 × 약 3파일 = ~39개+ 파일

- **특징:** 각 모듈이 완전히 독립. 한 모듈 에러가 다른 모듈에 영향 없음
- **약점:** 모듈 간 연동 시 수동 통합 필요. 공통 코드 변경 시 전체 확인 필요

```
agents_homepage/
├── app/                    ← Next.js 라우팅 (13개 페이지)
│   ├── layout.js           ← 전체 레이아웃 (Header + Footer 자동 적용)
│   ├── page.js             ← 홈 페이지
│   ├── products/
│   │   ├── page.js
│   │   └── [id]/page.js
│   ├── cart/page.js
│   ├── checkout/
│   ├── auth/
│   ├── mypage/page.js
│   ├── about/page.js
│   ├── admin/page.js
│   ├── contact/page.js
│   └── notice/page.js
├── components/             ← 재사용 컴포넌트 (23개)
│   ├── ui/                 ← 범용 UI (Button, Input, Card, Badge, Modal, Toast 등)
│   ├── layout/             ← 레이아웃 (Header, Footer, MobileMenu, Container)
│   ├── product/            ← 상품 (ProductCard, ProductGrid, ProductFilter 등)
│   └── home/               ← 홈 전용 (HeroBanner, PopularProducts 등)
├── contexts/               ← 전역 상태 (4개)
│   ├── CartContext.js
│   ├── AuthContext.js
│   ├── WishlistContext.js
│   └── ThemeContext.js
├── hooks/                  ← 커스텀 훅
├── lib/                    ← 중앙 데이터
│   ├── products.js         ← 상품 데이터 12개
│   └── utils.js            ← 유틸리티 함수
└── tailwind.config.js      ← 디자인 시스템 정의
```

- **특징:** 컴포넌트 재사용 극대화. Context API로 전역 상태 자연스럽게 공유
- **약점:** 프레임워크 의존도 높음. 프레임워크 이해 없이 수정 어려움

### 아키텍처 핵심 차이

| 관점 | module_homepage | agents_homepage |
|------|----------------|-----------------|
| **데이터 흐름** | 각 모듈이 독립적으로 Firestore 호출 | Context API로 중앙 집중 관리 |
| **UI 재사용** | HTML 복붙 (재사용 어려움) | React 컴포넌트 (props로 변형) |
| **레이아웃 공유** | common.js로 헤더/푸터 주입 | layout.js로 자동 적용 |
| **연동 방식** | 독립 개발 → 사후 연동 (충돌 위험) | 처음부터 연결된 구조 |
| **디자인 일관성** | common.css + CSS 변수 (사후 통합) | tailwind.config.js (사전 정의) |

| # | 반복 실수 | rndpark (1차) | module_homepage (2차) | agents_homepage (3차) | 판정 |
|---|----------|--------------|----------------------|----------------------|------|
| 1 | **배포 설정 반복 실패** | Google Drive 경로 + firebase.json 혼동 → 배포마다 10분+ | 인증 만료 + site 누락 → 배포 3~4회 반복 | Vercel 자동 배포 → 설정 자체 불필요 | **3차에서 완전 해소** |
| 2 | **브라우저 캐시 미반영** | 수정 후 화면 미반영 → 매번 혼란 | CSS 적용 안 됨 → 캐시+우선순위 문제 | Next.js 핫 리로드 + Vercel 캐시 무효화 | **3차에서 크게 개선** |
| 3 | **CSS 하드코딩** | 인라인 style 남용 → 6개 파일 수정 | 하드코딩 색상 → 13개 모듈 일괄 변환 | Tailwind 커스텀 테마 처음부터 적용 → 하드코딩 0건 | **3차에서 완전 예방** |
| 4 | **컨텍스트 소진** | 토큰 한계 → 맥락 재설명 부담 | 13개 모듈 → 컨텍스트 초과 | Claude Code 도구 기반 + 자동 압축 | **3차에서 크게 개선** |

### 분석

- **완전 해소 2건:** 배포 설정, CSS 하드코딩 — 기술 스택 전환과 교훈 적용으로 문제 자체가 사라짐
- **크게 개선 2건:** 브라우저 캐시, 컨텍스트 소진 — 발생 빈도 크게 감소했으나 본질적 한계는 존재
- 1차→2차에서 반복되던 4개 실수가, 2차→3차에서는 **반복 0건** 달성

## 11. 결론 및 다음 프로젝트 제언

### 결론

agents_homepage는 module_homepage의 교훈을 **기술 스택 전환**과 **개발 도구 업그레이드**로 한 단계 진화시킨 프로젝트입니다.

**핵심 성과:**

1. **병목 6개 중 5개 구조적 해소** — Firebase → Next.js+Vercel 전환으로 배포·인덱스·인증·연동 문제 소멸
2. **반복 실수 4개 중 4개 해소/개선** — 1차→2차에서 반복되던 패턴이 3차에서 최초로 끊김
3. **코드 품질 상승** — 컴포넌트 재사용, 디자인 시스템, 접근성, SEO까지 커버
4. **개발 효율 대폭 향상** — 파일 관리·Git·배포 자동화로 비생산적 시간 최소화

**아쉬운 점:**

1. **기능 범위 축소** — module_homepage(13모듈)에 비해 가격비교·PC견적·배송·프로모션·멤버십 미포함
2. **실서비스 백엔드 부재** — localStorage 기반이라 실제 서비스 운영에는 백엔드 추가 필요
3. **컨텍스트 소진 완전 해소 못함** — 개선되었으나 대규모 작업 시 여전히 한계 존재
